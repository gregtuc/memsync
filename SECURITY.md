# Security

memsync's trust boundary is the Git vault. Claude Code and Codex already keep
their own memory as local plaintext; memsync leaves those files untouched. Any
memory record that memsync writes to the vault is encrypted before it reaches
the vault working tree.

## Data locations

| Data | Local machine | Git vault and remote |
|---|---|---|
| Claude Code and Codex memory files | Plaintext, owned by the agent tool | Never copied as native files |
| memsync record contents | Plaintext only while being processed in memory | AES-256-GCM ciphertext (`*.enc`) |
| Vault key | `~/.config/memsync/key`, directory `0700`, file `0600` | Never committed to the Git vault |
| Device identity | `~/.config/memsync/device-id`, file `0600` | Included only inside encrypted records |
| Unfinished pairing state | `~/.config/memsync/join-state.json`, file `0600`, removed after success | Never leaves the joining laptop |
| Vault control data | Local Git metadata | The managed `.gitattributes` file and ordinary Git metadata are plaintext |

Every paired laptop stores the same vault key. Pairing transports that key from
an existing laptop to a new one inside an end-to-end encrypted reply. Thus the
key does leave the original laptop during an authorized pairing, but it does not
pass through GitHub or another Git remote in plaintext.

## Encryption envelope

Each record uses a 256-bit key generated by the operating system CSPRNG. Its
layout is:

```text
magic(8) || random nonce(12) || AES-256-GCM(ciphertext || tag)
```

A fresh random 96-bit nonce is generated for every encryption. Before rewriting
an existing record, memsync decrypts it locally and compares the plaintext. If
nothing changed, it leaves the existing ciphertext alone; this avoids routine
Git churn while encryption itself remains randomized.

AES-GCM authenticates the record contents and the envelope marker. A corrupt
record or one produced under another key fails to decrypt.

## Git guards

The vault installs `pre-commit`, `pre-merge-commit`, and `pre-push` guards through
Git's `core.hooksPath`. Before normal memsync-managed Git operations continue,
the guards check the index, working tree, and outgoing reachable history:

- every `*.enc` file must have a memsync envelope and decrypt under the local key;
- the only plaintext working-tree file allowed is the exact memsync-managed
  `.gitattributes` content;
- unresolved merge entries, foreign files, and corrupt ciphertext are rejected.

The pre-push guard walks every outgoing commit, so plaintext committed with
`--no-verify` and deleted later is still rejected. Incoming refs are checked
before a fetch can rebase or move the live working tree, and pairing validates
the full reachable history of a staged clone before activating it.

The guards are reinstalled during setup and repair because Git hooks are not
carried by a clone. They protect the supported workflow; a user or local attacker
who deliberately bypasses Git hooks is outside this guarantee.

## Pairing and authenticity

Pairing creates a temporary X25519 key pair on the joining laptop. Until the
join succeeds, its private half and the encrypted reply are kept in a private
local state file so a sign-in or network failure can resume without repeating
the exchange. The state file is deleted after success and never leaves that
laptop. The existing laptop derives a wrapping key for the public invite and
seals the vault key and remote URL with AES-GCM. Only the holder of the temporary
private key can open the reply.

The invite is public, but **public does not mean self-authenticating**. If an
attacker can replace it before it reaches the existing laptop, that laptop can
be tricked into sealing the vault key for the attacker. memsync therefore shows
a short fingerprint on both laptops and requires confirmation before sealing.
Compare the codes directly or over a separate authenticated channel, and cancel
if they differ. `memsync pair --yes` bypasses this interactive check and should
be used only when the invite was authenticated by another mechanism. Treat both
tokens as one-time setup material and do not reuse them.

Pairing also does not grant Git repository access. Each laptop must authenticate
to the private remote separately. memsync rejects HTTP(S) remote URLs with
embedded usernames, passwords, or tokens; use a credential helper or SSH so
credentials do not enter pairing payloads, Git config, or diagnostics.

## What the remote can observe

The remote receives encrypted record blobs, the managed `.gitattributes` file,
and normal Git history. A remote operator or repository reader can observe:

- repository and branch names;
- commit messages, timestamps, update cadence, and record counts;
- which encrypted files changed between commits;
- equality of ciphertext only when the unchanged file itself is reused.

Vault repositories use a local generic `memsync <memsync@localhost.invalid>`
Git identity and disable commit signing, so ordinary vault history does not
publish the laptop owner's global Git name, email, or signing identity.

They cannot decrypt record contents without a paired laptop's key. Use a private
repository anyway: encryption is the content boundary, while repository privacy
reduces metadata exposure and accidental access.

## Local and operational limits

- A process that can read the local agent memory or memsync key can read the same
  information memsync can. memsync does not defend a compromised laptop.
- Injected memories are instructions/context consumed by another model. Content
  originating in memory can therefore influence that model; review native memory
  stores if untrusted material may have entered them.
- Git encryption provides confidentiality and integrity, not availability or
  rollback detection. Someone controlling the remote can delete history or serve
  an older valid vault.
- Offline sessions use the last local encrypted cache. Until a later successful
  sync, another laptop will not receive the new records.
- If every paired copy of the key is lost, the remote ciphertext is unrecoverable.
  Back up the key only to storage that provides equivalent access controls.

## Reporting a vulnerability

Open a private security advisory on the GitHub repository. Please do not include
real memory contents, keys, pairing replies, or credential-bearing remote URLs in
an issue or diagnostic transcript.
